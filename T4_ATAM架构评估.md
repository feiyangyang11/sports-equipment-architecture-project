# T4 ATAM 架构评估

## 1. 评估目标

ATAM（Architecture Tradeoff Analysis Method）强调从业务驱动和质量属性出发，对架构决策进行系统评估。本次评估目标为：

1. 检查 T3 中的架构设计是否能够支撑核心业务与关键质量属性。
2. 识别影响质量属性的敏感点。
3. 识别多个质量属性之间的权衡点。
4. 区分有风险决策与无风险决策，并提出改进建议。

## 2. 评估对象

评估对象为 T3 中形成的总体架构方案，核心包括：

- 前后端分离的 B/S 架构
- RESTful API 接口风格
- Controller-Service-Repository 分层架构
- 基于事务的库存一致性控制
- 基于角色的权限控制
- 操作日志审计机制
- 单体部署 + MySQL 持久化方案

## 3. 业务驱动与质量属性复核

### 3.1 业务驱动

| 编号 | 业务驱动 |
| --- | --- |
| BD-01 | 学生需要快速查询可借器材并完成预约 |
| BD-02 | 管理员需要准确审核预约并办理借出归还 |
| BD-03 | 系统必须保证库存和借还状态一致 |
| BD-04 | 管理行为需要可追踪，便于责任认定与复盘 |
| BD-05 | 课程项目需要结构清晰、便于实现、展示与答辩 |

### 3.2 重点质量属性

| 优先级 | 质量属性 | 关注原因 |
| --- | --- | --- |
| 高 | 数据一致性 | 是系统核心正确性的基础 |
| 高 | 安全性 | 涉及角色权限和关键业务操作 |
| 高 | 性能 | 集中预约场景下用户体验敏感 |
| 中 | 可维护性 | 后续仍需实现、测试和扩展 |
| 中 | 可用性 | 系统应在实验环境中稳定运行 |
| 中 | 可审计性 | 管理场景存在责任追踪要求 |

## 4. ATAM 分析过程

本次评估参考 ATAM 的核心思路，按如下步骤开展：

1. 明确业务目标和关键质量属性。
2. 展示 T3 架构方案及其关键设计决策。
3. 建立质量属性效用树（Utility Tree）。
4. 围绕关键场景分析架构响应方式。
5. 识别敏感点、权衡点、风险点与非风险决策。

## 5. 质量属性效用树

| 质量属性 | 场景摘要 | 重要性 | 难度 |
| --- | --- | --- | --- |
| 数据一致性 | 并发预约同一器材时库存不得超卖 | 高 | 高 |
| 安全性 | 学生不得访问管理员接口 | 高 | 中 |
| 性能 | 高峰查询响应时间应控制在 2~3 秒内 | 高 | 中 |
| 可维护性 | 新增一个器材种类或统计接口时尽量局部修改 | 中 | 中 |
| 可用性 | 服务异常后 5 分钟内恢复 | 中 | 低 |
| 可审计性 | 审核、借出、归还、库存变更均应可追踪 | 中 | 低 |

## 6. 架构方案对关键场景的响应分析

### 6.1 场景 A：并发预约导致库存冲突

架构响应方式：

- 预约与借出核心逻辑集中在 Service 层。
- 借出办理放在数据库事务中执行。
- 库存与预约状态的变更由同一事务统一提交。
- 关键表通过主键、外键和索引约束业务一致性。

评估：

- 如果事务边界清晰，该方案可以有效避免“库存为负”。
- 如果仅在前端或接口层校验库存，而没有在事务内二次校验，则存在超卖风险。

### 6.2 场景 B：学生尝试越权访问管理员接口

架构响应方式：

- 接口层设置统一身份认证入口。
- 按角色执行 RBAC 权限判断。
- 对越权访问进行拒绝并记录日志。

评估：

- 该决策能够很好满足学生/管理员角色分离要求。
- 若权限校验散落在各 Controller 中，而非统一策略，则后续维护中容易遗漏。

### 6.3 场景 C：高峰期大量查询器材

架构响应方式：

- 使用 RESTful 查询接口和关系型数据库索引。
- 对查询和写操作进行职责分离。
- 前端只请求必要字段，减少冗余交互。

评估：

- 对课程项目规模而言，该方案基本足够。
- 若后续并发量显著提升，单体架构和无缓存方案会成为瓶颈。

### 6.4 场景 D：后续新增角色或新增统计报表

架构响应方式：

- 采用分层架构和标准化接口。
- 用户角色、业务逻辑、数据访问职责分离。

评估：

- 对新增统计接口、字段扩展、角色扩展较友好。
- 如果 DTO、Service、Repository 命名和边界不规范，扩展成本仍会上升。

## 7. 敏感点分析

敏感点是对某些质量属性影响特别大的架构参数或设计选择。

| 编号 | 敏感点 | 影响的质量属性 | 说明 |
| --- | --- | --- | --- |
| SP-01 | 借出与归还事务边界 | 数据一致性、性能 | 事务范围过小会导致状态不一致，过大则影响吞吐量 |
| SP-02 | 库存校验位置 | 数据一致性 | 若只做前置校验而不在事务内二次校验，容易超借 |
| SP-03 | 权限控制实现方式 | 安全性、可维护性 | 分散式鉴权容易遗漏；统一鉴权更稳健 |
| SP-04 | 预约与借用状态模型复杂度 | 可维护性、正确性 | 状态过少无法表达业务，状态过多又会增加实现复杂度 |
| SP-05 | 查询接口的字段粒度与数据库索引 | 性能 | 字段过多、索引不足会直接影响高峰查询性能 |
| SP-06 | 操作日志记录范围 | 可审计性、性能 | 日志过少无法追溯，日志过细可能增加存储与性能开销 |

## 8. 权衡点分析

权衡点是一个设计选择同时影响多个质量属性，需要在这些属性之间平衡。

| 编号 | 权衡点 | 涉及质量属性 | 分析 |
| --- | --- | --- | --- |
| TP-01 | 严格事务控制 | 数据一致性 vs 性能 | 强事务能保证库存正确，但会增加并发写压力 |
| TP-02 | 统一鉴权与拦截 | 安全性 vs 实现复杂度 | 统一鉴权更安全，但需要额外设计认证与拦截机制 |
| TP-03 | 分层架构 | 可维护性 vs 开发效率 | 分层提高结构清晰度，但初期代码量更多 |
| TP-04 | 完整日志留痕 | 可审计性 vs 性能/存储 | 日志越完整，追溯能力越强，但存储和写入成本越高 |
| TP-05 | 单体部署 | 可部署性/实现效率 vs 可扩展性/可用性 | 单体方案简单直接，但后续横向扩展能力较弱 |

## 9. 有风险决策分析

| 编号 | 有风险决策 | 风险说明 | 建议 |
| --- | --- | --- | --- |
| RD-01 | 采用单体部署且无缓存机制 | 当预约高峰显著增加时，查询和写操作都可能集中压到单实例和单数据库 | 在后续 T5/T6 可通过数据库索引、分页和只读查询优化先缓解，必要时再考虑缓存 |
| RD-02 | 如果库存校验只在预约创建阶段完成 | 借出办理时库存可能已变化，存在状态不一致风险 | 借出事务中必须再次校验并更新库存 |
| RD-03 | 如果权限判断散落在各个 Controller 中 | 后续扩展接口时容易遗漏鉴权，形成越权漏洞 | 采用统一认证过滤器或中间件式鉴权 |
| RD-04 | 如果未设计预约/借用状态机约束 | 容易出现“已取消仍可借出”“已归还重复归还”等非法状态 | 在 Service 层统一定义允许的状态迁移 |
| RD-05 | 如果异常登记与归还流程分离且不在同一事务中 | 可能出现库存已回收但损坏信息未记录的情况 | 归还和异常登记在事务内统一提交 |

## 10. 无风险决策分析

| 编号 | 无风险决策 | 原因 |
| --- | --- | --- |
| NRD-01 | 采用前后端分离 + RESTful API | 结构清晰，适合课程项目实现与展示 |
| NRD-02 | 采用 Controller-Service-Repository 分层 | 能较好支撑可维护性和后续扩展 |
| NRD-03 | 使用关系型数据库管理核心业务数据 | 与器材、预约、借还、日志等强一致关联数据高度匹配 |
| NRD-04 | 引入 RBAC 角色控制 | 与学生/管理员双角色场景天然匹配 |
| NRD-05 | 建立操作日志机制 | 对管理系统是必要且稳定收益明显的决策 |

## 11. ATAM 结论

综合评估后，可以得出以下结论：

1. T3 给出的架构总体上能够满足课程项目在业务完整性、结构清晰性和实现可行性上的要求。
2. 该架构对“数据一致性、安全性、可维护性”三个核心目标支撑较好。
3. 当前最需要重点把控的不是技术栈本身，而是事务边界、库存校验位置、状态流转规则和统一鉴权实现。
4. 在课程项目规模内，单体架构是合理选择，但它在高并发、高可用和大规模扩展方面存在天然上限。

## 12. 改进建议

为降低风险、提升后续实现质量，建议在 T5 阶段优先落实以下事项：

1. 为 `equipment`、`reservation`、`borrow_record` 等核心表建立必要索引。
2. 在借出与归还服务中明确事务边界，并做二次库存校验。
3. 统一封装鉴权与角色检查逻辑，避免散落实现。
4. 将预约和借用状态迁移规则固化为明确的状态检查代码。
5. 为日志、异常和库存变更编写基础集成测试，提前验证关键质量属性。
