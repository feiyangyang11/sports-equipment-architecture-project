# T1 软件需求分析

## 1. 文档说明

本文档描述当前仓库 `version 1.0` 已实现系统的需求范围，作为 T2 用例分析、T3 架构设计和 T6 系统测试的统一基线。需求内容以现有代码、数据库脚本和前端页面为准，不把尚未实现的设想写入本版本验收范围。

## 2. 项目背景

高校体育器材通常由场馆管理员统一管理，日常业务包括器材查询、学生预约、管理员审核、借出和归还。人工登记容易出现预约状态不清晰、库存变化不同步、记录难以查询等问题。

本项目实现一个基于 C++ 的体育器材预约与借还管理系统，通过浏览器页面和 REST API，将以下业务串成完整闭环：

`登录 -> 查询器材 -> 提交预约 -> 审核预约 -> 办理借出 -> 办理归还`

## 3. 建设目标

1. 为学生提供器材查询、预约申请、预约记录和借用记录查询能力。
2. 为管理员提供预约审核、借出办理和归还办理能力。
3. 通过明确的状态流转约束非法操作。
4. 在借出和归还时同步修改业务记录与库存，保证核心数据一致。
5. 使用清晰的前后端分离和后端分层结构，便于课程展示、测试和维护。

## 4. 参与者

| 参与者 | 职责与关注点 |
| --- | --- |
| 学生用户 | 登录、查询器材、提交或取消预约、查看个人预约和借用记录 |
| 管理员 | 登录、查询全部预约、审核预约、办理借出和归还、查询借用记录 |
| 系统 | 身份校验、角色检查、状态校验、库存更新、数据持久化 |

## 5. 系统范围

### 5.1 本版本包含

- 用户登录、当前用户查询和退出登录
- 学生与管理员角色识别
- 器材分类查询
- 器材分页查询、按分类筛选和详情查询
- 学生提交预约、查询个人预约、查看详情和取消预约
- 管理员分页查询预约、审核通过和审核拒绝
- 管理员根据已通过预约办理借出
- 管理员办理归还
- 学生查询个人借用记录
- 管理员查询全部借用记录
- 借出和归还过程中的库存同步更新
- 学生端和管理员端静态网页

### 5.2 本版本不包含

- 器材和分类的新增、修改、删除
- 自动逾期扫描和逾期处理接口
- 独立统计报表
- 操作日志的业务写入和查询接口
- 校园统一身份认证、消息推送和多校区部署

数据库中保留了 `operation_log` 表，借用状态中保留了 `OVERDUE/CLOSED` 枚举，作为后续版本扩展位，不属于当前 T5/T6 验收功能。

## 6. 功能性需求

| 编号 | 模块 | 需求描述 |
| --- | --- | --- |
| FR-01 | 健康检查 | 系统应提供健康检查接口，用于确认后端服务可访问 |
| FR-02 | 用户认证 | 用户应能使用用户名和密码登录、查询当前用户并退出登录 |
| FR-03 | 角色权限 | 学生接口只能由学生访问，管理员接口只能由管理员访问 |
| FR-04 | 器材分类查询 | 系统应返回当前启用的器材分类 |
| FR-05 | 器材查询 | 系统应支持器材分页查询、分类筛选和详情查询 |
| FR-06 | 创建预约 | 学生应能选择器材、预约时间段和数量提交预约 |
| FR-07 | 学生预约查询 | 学生应能分页查询自己的预约、按状态筛选并查看详情 |
| FR-08 | 取消预约 | 学生应能取消自己的 `PENDING` 或 `APPROVED` 预约 |
| FR-09 | 预约审核 | 管理员应能查询全部预约，并通过或拒绝 `PENDING` 预约 |
| FR-10 | 办理借出 | 管理员应能将 `APPROVED` 预约办理为借用记录并扣减库存 |
| FR-11 | 借用查询 | 学生应能查询自己的借用记录，管理员应能查询全部借用记录 |
| FR-12 | 办理归还 | 管理员应能归还 `BORROWING` 记录、恢复库存并完成预约 |
| FR-13 | 前端交互 | 系统应提供登录页、学生业务页和管理员业务页完成主要流程 |

### 6.1 用户认证

- 登录成功后返回随机 Token 和用户基本信息。
- 后续受保护接口使用 `Authorization: Bearer <token>`。
- 当前 Oat++ Endpoint 将 `Authorization` 声明为必填 Header：完全缺少该 Header 时由框架返回 400，Header 存在但 Token 无效时由业务代码返回 401。
- 用户退出登录后，原 Token 立即失效。
- 当前 Token 保存在后端进程内存中，后端重启后需要重新登录。

### 6.2 器材查询

- 分类和器材查询为公开只读接口。
- 器材列表支持 `pageNo`、`pageSize` 和可选的 `categoryId`。
- `pageNo >= 1`，`pageSize` 范围为 `1..100`。
- 器材详情包含分类、编号、名称、规格、存放位置、总库存、可用库存和状态。

### 6.3 预约管理

- 只有学生可以创建预约。
- 数量必须大于 0。
- 时间格式必须为 `YYYY-MM-DD HH:MM:SS`，结束时间晚于开始时间。
- 器材必须存在且状态为 `AVAILABLE`。
- 申请数量不得超过当前可用库存，也不得导致相同时段预约总量超过可用库存。
- 新预约状态为 `PENDING`。
- 学生只能查询和取消自己的预约。

### 6.4 预约审核

- 只有管理员可以查看管理员预约列表并执行审核。
- 只有 `PENDING` 预约可以变为 `APPROVED` 或 `REJECTED`。
- 审核通过前再次校验器材状态和预约时间段内的数量。
- 重复审核应被拒绝。

### 6.5 借出与归还

- 只有 `APPROVED` 预约可以办理借出。
- 借出时间和应还时间格式必须合法，应还时间晚于借出时间。
- 借出成功后创建 `BORROWING` 借用记录、预约变为 `BORROWED`、可用库存扣减。
- 只有 `BORROWING` 记录可以办理归还。
- 归还成功后借用记录变为 `RETURNED`、预约变为 `COMPLETED`、可用库存恢复。
- 借出和归还中的多表修改必须分别在单个数据库事务中完成。

## 7. 非功能性需求

| 编号 | 类别 | 当前版本要求 |
| --- | --- | --- |
| NFR-01 | 性能 | 本地课程演示环境下，200 个集中器材查询请求的平均响应时间不超过 2 秒，P95 不超过 3 秒，请求成功率为 100% |
| NFR-02 | 数据一致性 | 并发办理同一预约时最多生成一条借用记录；库存不得为负；借出扣减和归还恢复必须与状态更新一致 |
| NFR-03 | 安全性 | 缺少或使用无效 Token 时不能访问受保护接口；学生不能访问管理员接口；管理员不能以学生身份创建预约 |
| NFR-04 | 可恢复性 | 后端进程停止后应能在 5 分钟内重新启动；已提交业务数据应保留；用户重新登录后可继续使用 |
| NFR-05 | 可维护性 | HTTP、业务规则和 SQL 分别位于 Controller、Service 和 DAO；前端通过统一 API 模块调用后端 |
| NFR-06 | 易用性 | 用户可通过浏览器完成主要业务流程，页面应提供状态、空数据和错误提示 |
| NFR-07 | 兼容性 | 前端面向现代 Edge/Chrome；后端使用标准 HTTP/JSON 接口 |
| NFR-08 | 可部署性 | Windows 环境下可使用 CMake 构建后端，并通过仓库脚本启动后端和静态前端 |

## 8. 业务状态与规则

### 8.1 预约状态

```text
PENDING ----approve----> APPROVED ----borrow----> BORROWED ----return----> COMPLETED
   |                         |
   +----reject----> REJECTED +----cancel----> CANCELED
   |
   +----cancel----> CANCELED
```

### 8.2 借用状态

当前实现使用的主流程为：

```text
BORROWING ----return----> RETURNED
```

`OVERDUE` 和 `CLOSED` 仅为数据模型预留状态。

### 8.3 核心业务规则

| 编号 | 规则 |
| --- | --- |
| BR-01 | 学生和管理员只能调用符合自身角色的受保护接口 |
| BR-02 | 学生只能查看和取消自己的预约、借用记录 |
| BR-03 | 只有 `AVAILABLE` 器材可以预约 |
| BR-04 | 只有 `PENDING` 预约可以审核 |
| BR-05 | 只有 `PENDING/APPROVED` 预约可以取消 |
| BR-06 | 只有 `APPROVED` 预约可以借出 |
| BR-07 | 同一预约最多对应一条借用记录 |
| BR-08 | 只有 `BORROWING` 借用记录可以归还 |
| BR-09 | 借出和归还必须保证预约、借用记录和库存的一致性 |

## 9. 关键数据对象

| 数据对象 | 当前用途 |
| --- | --- |
| User | 用户账号、密码摘要、角色和状态 |
| EquipmentCategory | 器材分类 |
| Equipment | 器材信息、总库存、可用库存和状态 |
| Reservation | 预约时间、数量、审核信息和状态 |
| BorrowRecord | 借出、应还、归还信息和借用状态 |

## 10. 验收边界

当前版本以核心链路能够通过真实前后端和 MySQL 完成作为 T5 实现验收标准；T6 应按照本文档 FR-01～FR-13 和 NFR-01～NFR-08 进行测试。范围外功能不作为本版本失败项。
