# T1 软件需求分析

## 1. 项目背景

高校体育器材通常由体育部或场馆管理员统一管理，日常业务包括器材信息维护、库存统计、学生预约、借出归还以及逾期处理。传统人工登记方式存在以下问题：

- 器材库存变化不及时，容易出现“明明已经借完却仍能登记”的情况。
- 预约申请、审批、借出、归还之间缺少统一状态流转，责任界限不清晰。
- 管理员需要频繁手工核对库存和历史记录，工作量大且容易出错。
- 统计信息分散，不利于分析器材利用率、借用情况和高峰借用时段。

因此，本项目拟设计并实现一个“基于 C++ 的体育器材预约与借还管理系统”，服务于校园体育器材管理的典型业务场景。

## 2. 建设目标

系统目标如下：

1. 为学生提供统一的器材查询、预约、记录查看和归还入口。
2. 为管理员提供器材维护、预约审核、借出归还办理工具。
3. 建立统一的库存与借还状态模型，避免重复预约、超借和数据不一致。
4. 通过结构化数据与操作日志支撑后续统计分析、课程展示与系统扩展。

## 3. 相关干系人

| 干系人 | 关注点 |
| --- | --- |
| 学生用户 | 器材是否可借、预约是否成功、个人借还记录是否清晰 |
| 管理员 | 库存是否准确、流程是否顺畅、借还记录是否可追踪 |
| 体育部门 | 器材利用率、借用情况、管理效率、数据留痕 |
| 开发与维护人员 | 系统结构是否清晰、需求是否明确、是否便于扩展和调试 |

## 4. 系统范围

系统覆盖的业务范围：

- 用户登录与角色识别
- 体育器材分类、库存与状态管理
- 学生预约提交、取消和查询
- 管理员审核预约
- 借出与归还办理
- 逾期处理与跟踪
- 使用记录、库存状态与统计查询
- 管理员关键操作日志

系统暂不覆盖的内容：

- 第三方支付
- 校园统一身份认证深度对接
- 多校区分布式部署
- 短信、邮件、微信等外部消息推送

## 5. 功能性需求

### 5.1 功能模块总览

| 模块编号 | 模块名称 | 主要职责 |
| --- | --- | --- |
| FR-01 | 用户认证与角色识别 | 登录、身份校验、角色区分 |
| FR-02 | 器材查询 | 查询器材分类、名称、库存、状态、可借数量 |
| FR-03 | 预约管理 | 提交预约、取消预约、查看预约状态 |
| FR-04 | 预约审核 | 管理员通过或拒绝预约申请 |
| FR-05 | 借出管理 | 办理借出并生成借用记录 |
| FR-06 | 归还管理 | 办理归还并回收库存 |
| FR-07 | 逾期管理 | 标记逾期、记录备注说明 |
| FR-08 | 器材管理 | 新增器材、修改器材信息、调整库存与状态 |
| FR-09 | 统计查询 | 查询借用记录、器材使用次数、库存状态 |
| FR-10 | 操作日志 | 记录管理员关键操作 |

### 5.2 详细功能需求

#### FR-01 用户认证与角色识别

系统应支持学生用户与管理员通过用户名和密码登录。登录成功后，系统需要识别当前用户角色，并根据角色返回不同的功能菜单与操作权限。

#### FR-02 器材查询

学生与管理员均可查看器材列表，查询内容至少包括器材名称、分类、总库存、可用库存、当前状态和可预约时间信息。系统应支持按分类、名称、状态进行筛选。

#### FR-03 预约管理

学生可以在库存充足且器材状态允许的前提下提交预约申请，并指定预约时间段与数量。学生可查看个人预约记录，并在管理员尚未办理借出前取消预约。

#### FR-04 预约审核

管理员可以查看待审核预约，对申请进行通过或拒绝处理，并填写审核意见。审核结果应反馈给学生，并写入预约状态历史。

#### FR-05 借出管理

对于审核通过的预约，管理员可办理借出。借出成功后，系统应生成借用记录、更新可用库存，并将预约状态变更为“已借出”。

#### FR-06 归还管理

管理员在学生归还器材时进行确认。归还成功后，系统应更新借用记录、回收库存，并将该次借用状态标记为“已归还”。

#### FR-07 逾期管理

管理员可对借用记录标记逾期情况，并保存说明备注，方便后续查询与追踪。

#### FR-08 器材管理

管理员可新增器材类别和器材信息，维护器材名称、分类、库存数量、状态、存放位置等基础数据；支持对器材进行启用、停用、维修中等状态调整。

#### FR-09 统计查询

系统应提供按器材、时间范围、状态、用户维度的记录查询能力，并支持展示器材使用次数、当前库存状态、借还历史等统计结果。

#### FR-10 操作日志

对于登录、预约审核、借出、归还、库存调整等关键管理操作，系统应记录操作人、操作时间、操作类型、关键对象和结果。

## 6. 非功能性需求

| 编号 | 类别 | 需求描述 |
| --- | --- | --- |
| NFR-01 | 性能 | 在 200 名学生短时集中查询器材时，普通查询平均响应时间不超过 2 秒，95% 请求不超过 3 秒 |
| NFR-02 | 数据一致性 | 同一器材在并发预约和借出场景下不得出现可用库存为负数、重复借出或状态错乱 |
| NFR-03 | 可用性 | 系统出现进程异常后，应能在 5 分钟内完成重启并继续提供核心查询与管理功能 |
| NFR-04 | 安全性 | 未认证用户不能访问业务接口；普通学生不能调用管理员接口；关键数据传输需经过身份校验 |
| NFR-05 | 可维护性 | 业务逻辑、接口层、数据访问层应解耦，新增一个器材分类或统计接口时，不应影响主要业务流程 |
| NFR-06 | 可扩展性 | 后续支持新增角色、接入消息提醒或扩充器材种类时，核心架构无需整体推翻 |
| NFR-07 | 易用性 | 学生应能在较少操作步骤内完成器材查询和预约，管理员应能通过表格与按钮完成主要业务处理 |
| NFR-08 | 可审计性 | 系统应保留关键管理操作日志，支持事后追溯“谁在什么时间做了什么操作” |
| NFR-09 | 兼容性 | 系统前端应兼容常见桌面浏览器，后端应支持标准 RESTful 接口调用 |
| NFR-10 | 可部署性 | 系统应能在课程实验环境下通过 CMake 编译部署，并使用 MySQL 作为持久化存储 |

## 7. 业务规则与合理化假设

为使需求能够直接落地到后续设计，给出如下业务规则：

| 编号 | 规则内容 |
| --- | --- |
| BR-01 | 只有学生用户可以提交预约，管理员不能以学生身份创建预约 |
| BR-02 | 只有状态为“可用”的器材允许预约和借出 |
| BR-03 | 预约申请必须经过管理员审核后才能进入借出流程 |
| BR-04 | 同一时段内，同一器材的已通过预约数量与已借出数量总和不能超过可用库存 |
| BR-05 | 已拒绝、已取消或已完成的预约不能再次办理借出 |
| BR-06 | 归还后库存应立即回收，并同步更新借用记录状态 |
| BR-07 | 管理员对库存和器材状态的调整必须记录到操作日志 |
| BR-08 | 统计查询以历史记录为准，不应直接依赖前端缓存数据 |

## 8. 关键数据对象

| 数据对象 | 说明 |
| --- | --- |
| User | 学生与管理员基本信息、账号、密码、角色 |
| EquipmentCategory | 器材分类，如球类、球拍类 |
| Equipment | 器材基础信息、库存、可用数量、状态 |
| Reservation | 学生预约记录、预约时间段、审核状态 |
| BorrowRecord | 借出与归还记录、借出时间、归还时间、状态 |
| OperationLog | 管理员关键操作日志 |

## 9. 需求总结

本项目满足课程中“中等复杂规模软件系统”的定位，既包含较完整的业务流程，也包含性能、安全、一致性、可维护性等典型架构关注点。T1 的输出为后续 T2 用例分析、T3 架构设计与 T4 架构评估提供统一输入。
