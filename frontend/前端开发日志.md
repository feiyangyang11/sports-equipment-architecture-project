# 前端开发日志

## 1. 这份日志的目标

这份文档不记录功能测试过程，而是重点记录这次前端开发时的思路、取舍、落地方式和中途调整。

我希望它能回答这几个问题：

1. 为什么前端先这样设计，而不是直接随便写几个页面。
2. 页面、脚本、状态、接口适配层分别应该放在哪里。
3. 当开发过程中出现更好的思路时，为什么要改，以及改动具体体现在哪些代码里。

---

## 2. 初始约束与设计边界

在开始写前端前，项目已经具备了比较完整的后端接口：

- `auth`
  - `POST /api/login`
  - `GET /api/me`
  - `POST /api/logout`
- `equipment`
  - `GET /api/equipment`
  - `GET /api/equipment/{id}`
  - `GET /api/equipment-categories`
- `reservation`
  - 学生侧：创建、查询、取消
  - 管理员侧：分页审核、通过、拒绝
- `borrow`
  - 管理员办理借出、办理归还
  - 学生查询自己的借用
  - 管理员查询借用记录

但这次开发里，用户明确提出一个要求：

> 先不考虑前后端联调的事情，先把整个前端开发完成。

这句话会直接影响前端技术方案。

如果此时仍然坚持“页面一上来就只连真实后端”，会立刻遇到几个问题：

1. 页面开发速度被联调环境卡住。
2. 浏览器跨域、同源、后端运行目录等问题会抢走注意力。
3. 很多页面细节还没定下来，就被迫一边改 UI 一边排接口问题。

所以我一开始就决定：

**前端先以“静态多页面 + 默认 mock 数据模式”完成整体开发，接口层保持和真实后端一致，后续再切换到真实 API。**

## 3. 第一版方案：静态多页面 + Bootstrap + 原生 JS

### 3.1 为什么不用 Vue / React

这次我没有上框架，原因不是框架不好，而是当前项目更适合用：

- `HTML`
- `CSS`
- `JavaScript`
- `Bootstrap`

原因有 4 个：

1. 项目原始规划就是静态前端目录。
2. 你后面想借助这份日志理解开发过程，原生 JS 更容易看到“页面到底是怎么一步步拼起来的”。
3. 当前页面数量不算少，但交互复杂度还没有高到必须引入组件框架。
4. 课程项目里，先把“接口驱动的页面开发方式”搞明白，比先学一整套前端工程化更划算。

### 3.2 为什么不是单页，而是多页面

一开始有两个可选方案：

#### 方案 A：一个 `index.html`，所有内容都在一个页面里切换

优点：

- 文件少
- 看起来像 SPA

缺点：

- 角色切换、页面守卫、导航状态都要自己模拟
- 逻辑容易混成一团
- 对当前项目阶段来说，不够直观

#### 方案 B：按业务页面拆成多个 HTML

优点：

- 每个页面的职责清楚
- 更符合“登录页 / 学生页 / 管理员页”的真实结构
- 更容易和后端路由资源边界对应

最终我选了 **方案 B**。

所以现在前端结构按页面拆成了：

- `pages/login.html`
- `pages/student-equipment.html`
- `pages/student-reservations.html`
- `pages/student-borrows.html`
- `pages/admin-reservations.html`
- `pages/admin-borrows.html`

这一步对应的代码文件是：

- [student-equipment.html](/D:/newdir/sports-equipment-architecture-project/frontend/pages/student-equipment.html)

## 4. 第二版思路：页面不能各写各的，要先抽“公共层”

如果直接写 6 个页面，很快会出现一个问题：

- 每页都要登录态
- 每页都要判断角色
- 每页都要有导航
- 每页都要有退出登录
- 每页都要有状态徽章、分页、错误提示

如果这些东西散落在每个页面脚本里，后面会非常难维护。

所以在动页面逻辑之前，我先抽了 8 个公共文件：

- [config.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/config.js)
- [storage.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/storage.js)
- [auth.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/auth.js)
- [guard.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/guard.js)
- [api.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/api.js)
- [mock-data.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/mock-data.js)
- [mock-api.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/mock-api.js)
- [shell.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/shell.js)
- [utils.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/utils.js)
- [ui.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/ui.js)

### 4.1 `config.js`

这里主要做两件事：

1. 统一前端配置
2. 明确声明当前是否走 mock

关键点是：

```js
export const USE_MOCK = true;
```

这意味着：

- 当前默认不依赖真实后端联调
- 但并没有把未来切换真实接口的路堵死

### 4.2 `storage.js`

这是一个很小但很重要的抽象层。

它把：

- 读 `localStorage`
- 写 `localStorage`
- 删除值

统一封装起来，避免这些操作散落到每个页面脚本里。

### 4.3 `auth.js`

这个文件的职责是：

- 保存登录态
- 读取登录态
- 读取 token
- 读取当前用户
- 清空登录态

也就是说，它只做一件事：

**会话状态管理**

它不直接管网络请求，也不直接管页面跳转。

### 4.4 `guard.js`

这个文件专门负责页面访问控制。

我把它设计成：

- `requireRole("STUDENT")`
- `requireRole("ADMIN")`
- `redirectByRole(user)`

这样每个页面脚本一开头就能非常清楚地表达：

```js
const currentUser = requireRole("STUDENT");
```

这背后的好处是：

- 页面角色要求一眼可见
- 未登录用户会被拦截回登录页
- 登录错角色的用户会被重定向到正确页面

### 4.5 `shell.js`

这是这次前端里一个很关键的抽象。

我最开始有过一个更“直接”的想法：

> 让每个 HTML 自己写导航、标题区、侧栏和用户信息。

这个想法很快被我否掉了，因为：

- 学生端和管理员端都有侧边栏
- 头部区块结构基本一致
- 用户信息卡和退出登录按钮结构一致

如果每页都手写一套，重复会非常严重。

所以我后来改成了：

**所有业务页面都通过 `mountShell(...)` 统一生成页面壳层。**

这一改动对应的代码就是：

- [shell.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/shell.js)

它负责统一生成：

- 左侧导航栏
- 品牌区
- 用户信息区
- 页面头图区域
- 主内容区
- 退出登录逻辑

这一步是本次开发里的第一个重要“思路升级”。

## 5. 第三版思路：既然暂不联调，就把 mock 做成“真实业务替身”

### 5.1 最开始的简单思路

一开始可以很偷懒地写成：

- 页面里写死几条数组
- 每个页面只显示静态假数据

但我没有这样做，因为这样做虽然快，却有两个问题：

1. 页面之间没有共享状态
2. 无法模拟真实业务流转

比如：

- 学生创建了一条预约
- 管理员页看不到
- 管理员审核后借还页也看不到变化

这种“假页面”对学习价值很低。

### 5.2 后来改成的方案

我把 mock 设计成了：

**一个持久化在 `localStorage` 里的前端模拟业务数据库**

也就是：

- [mock-data.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/mock-data.js)
- [mock-api.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/mock-api.js)

#### `mock-data.js`

负责提供初始化数据：

- 用户
- 器材分类
- 器材
- 预约记录
- 借用记录
- 计数器

#### `mock-api.js`

负责模拟真实接口行为：

- 登录
- 获取当前用户
- 查询器材
- 创建预约
- 审核预约
- 办理借出
- 办理归还

而且这些操作会真实修改 mock 状态：

- 新增预约会写入 `reservations`
- 审核通过会改预约状态
- 办理借出会新增 `borrowRecords`
- 办理归还会同步更新借用、预约和库存

这一步的意义非常大：

**虽然暂时不联调，但页面之间已经不是“假联动”，而是有真实状态流转的。**

这也是本次开发里的第二个重要思路升级。

## 6. 第四版思路：页面脚本不直接写 `fetch`，而是统一走 `api.js`

我没有让页面脚本直接去调用：

```js
fetch(...)
```

而是做了一层：

- [api.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/api.js)

它的职责是：

1. 统一暴露业务 API
2. 当前走 mock 时调用 `mock-api.js`
3. 将来切真实接口时走 `fetch`

这意味着页面脚本永远只关心这种调用方式：

```js
await reservationApi.create(payload)
await borrowApi.getAdminPage(...)
await authApi.login(username, password)
```

页面不必知道：

- 当前是不是 mock
- token 怎么塞请求头
- fetch 怎么处理错误

这一步把“页面逻辑”和“数据来源”彻底解耦了。

## 7. 页面实现日志

下面开始按页面记录这次真正的实现思路。

### 7.1 登录页 `login.html + login.js`

对应文件：

- [login.html](/D:/newdir/sports-equipment-architecture-project/frontend/pages/login.html)
- [login.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/login.js)

#### 这一页的目标

不是只做一个表单，而是完成 3 件事：

1. 输入用户名密码
2. 调登录接口
3. 根据角色跳转到不同工作台

#### 为什么这样设计

登录页是整个前端唯一的公共入口，所以我让它承担“系统入口说明”的职责，而不只是冷冰冰的两个输入框。

因此页面左边是“故事区”，右边才是登录卡片。

这种布局的好处是：

- 第一次打开页面就知道系统在干什么
- 学生和管理员的主流程一开始就被说明清楚
- 后面你回看代码时，也更容易理解为什么会有角色分流

#### 编码实现重点

`login.js` 的主流程是：

1. 如果已登录，直接按角色跳转
2. 监听表单提交
3. 调 `authApi.login(...)`
4. 把 `token + user` 存到 `localStorage`
5. 调 `redirectByRole(user)`

这一页故意保持很薄，因为登录动作本身不该背太多页面逻辑。

### 7.2 学生器材页 `student-equipment.html + student-equipment.js`

对应文件：

- [student-equipment.html](/D:/newdir/sports-equipment-architecture-project/frontend/pages/student-equipment.html)
- [student-equipment.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/student-equipment.js)

#### 这一页要解决的问题

学生端最常走的入口，不是“我的预约”，而是先“看有什么器材可选”。

所以这页承担的是：

- 分类筛选
- 查看器材详情
- 发起预约

#### 页面结构设计

这页最终是：

- 顶部：页面说明和统计卡
- 左侧主区：器材列表
- 右侧侧栏：当前选中器材详情
- 弹窗：预约表单

#### 为什么不是“列表页 + 独立详情页”

一开始也可以做成：

- 列表页
- 点击跳到器材详情页

但我最后没有这样做，因为当前学生最关键的动作链是：

**浏览 -> 看明细 -> 立即预约**

所以把详情放进同页右侧栏，会更顺。

#### 编码实现重点

这一页的核心状态包括：

- `categories`
- `pageNo`
- `categoryId`
- `pageResult`
- `selectedEquipmentId`
- `modalEquipmentId`

也就是说，我没有把页面写成“点一下就随缘改 DOM”，而是先维护一个明确的页面状态对象，再由状态决定渲染结果。

这一步是为了让你后面更容易理解：

**复杂页面本质上是在维护一个小型前端状态机。**

#### 一个中途调整

最开始我本来只想在列表卡片上放“预约”按钮。  
但后来发现右侧详情区如果只有展示，没有动作，会显得断。

所以我追加了：

- 详情区里也提供“预约这件器材”按钮

这样用户无论从列表卡片还是详情区，都能进入预约动作。

这就是一个典型的“写着写着发现更顺的交互路径，于是同步改代码”的例子。

### 7.3 学生预约页 `student-reservations.html + student-reservations.js`

对应文件：

- [student-reservations.html](/D:/newdir/sports-equipment-architecture-project/frontend/pages/student-reservations.html)
- [student-reservations.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/student-reservations.js)

#### 这一页的目标

这页不是让学生“继续创建内容”，而是回答：

- 我预约过什么？
- 当前审核到哪一步了？
- 哪些预约还能取消？

#### 页面结构设计

最终结构是：

- 顶部统计卡
- 左侧：预约表格列表
- 右侧：当前选中预约详情
- 弹窗：取消预约

#### 为什么要做“列表 + 详情双栏”

如果只做表格，用户只能看到一堆行，很难真正理解某条预约的上下文。  
如果每次都跳详情页，又会太碎。

所以我用了和器材页类似的思路：

- 列表负责筛选和切换
- 侧栏负责解释单条记录

这能明显降低状态型页面的理解成本。

#### 编码实现重点

我把“能否取消预约”抽成了一个明确规则：

```js
["PENDING", "APPROVED"].includes(item.status)
```

这件事看似小，但它体现了一个很重要的前端思维：

**前端不只是渲染状态，还要把状态约束翻译成页面上的动作可用性。**

也就是：

- 不是所有记录都显示“取消”
- 只有符合条件的记录才显示

### 7.4 学生借用页 `student-borrows.html + student-borrows.js`

对应文件：

- [student-borrows.html](/D:/newdir/sports-equipment-architecture-project/frontend/pages/student-borrows.html)
- [student-borrows.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/student-borrows.js)

#### 这一页的目标

和预约页相比，这页更强调：

- 当前还借着什么
- 什么时候到期
- 是否已经归还

#### 页面定位上的一个关键区别

预约页是“申请中的状态流”  
借用页是“已经发生的事实记录”

所以这页比预约页更看重：

- 借出时间
- 应还时间
- 实际归还时间
- 当前借用状态

#### 编码实现重点

为了让学生对“已借出 / 已归还 / 逾期”的区别更敏感，我把状态筛选做在了页面上方，并单独做了统计卡。

这比一上来就堆更多字段更有效，因为页面一眼就能看出：

- 当前页有多少条借用
- 有多少条仍在借出中
- 有多少条已归还

### 7.5 管理员预约审核页 `admin-reservations.html + admin-reservations.js`

对应文件：

- [admin-reservations.html](/D:/newdir/sports-equipment-architecture-project/frontend/pages/admin-reservations.html)
- [admin-reservations.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/admin-reservations.js)

#### 这一页的目标

管理员页和学生页最大的区别不是“权限不同”这么简单，而是：

**管理员页更像工作台。**

所以这一页必须优先强调：

- 谁发起了申请
- 申请了什么
- 当前是否需要处理
- 我现在能做什么动作

#### 页面结构设计

这页延续了双栏结构，但列表区更“操作导向”：

- 行内直接给“通过 / 拒绝”
- 右侧详情给完整上下文

#### 一个重要的实现选择

我没有把“通过 / 拒绝”做成列表外的统一按钮，而是行内动作 + 详情区动作并存。

原因是：

- 管理员可能在扫表时快速处理
- 也可能先看详情，再做决定

这两种习惯都应该被支持。

#### 编码实现重点

这页用了一个通用审核弹窗：

- 同一套表单
- 通过时切 `approve`
- 拒绝时切 `reject`

也就是说，我没有写两个完全独立的 modal。  
这是一次刻意的收敛，让“审核动作”在页面层也形成一个统一抽象。

### 7.6 管理员借还页 `admin-borrows.html + admin-borrows.js`

对应文件：

- [admin-borrows.html](/D:/newdir/sports-equipment-architecture-project/frontend/pages/admin-borrows.html)
- [admin-borrows.js](/D:/newdir/sports-equipment-architecture-project/frontend/assets/js/admin-borrows.js)

#### 这一页的目标

这是整个前端里业务味最重的一页。

它不是简单列表页，而是要同时承接两件事：

1. 对已审核通过预约办理借出
2. 对已经借出的记录办理归还

#### 最开始的想法

一开始有个很自然的做法：

- 单独做一个“借出办理页”
- 再单独做一个“归还管理页”

这样拆当然可以，但我后来觉得不合适。

因为在真实业务里，管理员更像是在维护一张“借还工作台”，而不是在两个互不相关的页面之间来回跳。

#### 后来改成的方案

最终我把它设计成同一页的两段：

- 上半段：待办理借出的已通过预约
- 下半段：借用记录列表
- 右侧：当前选中借用详情

这就是本次开发里第三个比较大的思路调整。

#### 为什么这是更好的设计

因为管理员真实工作流更接近：

- 先看到有哪些已通过预约待借出
- 办理完借出后，记录立刻进入借用列表
- 后续再从借用列表里办理归还

把这几个阶段放在同一页，状态推进关系更清楚。

#### 编码实现重点

这一页同时调用了两组接口：

- `reservationApi.getAdminPage({ status: "APPROVED" })`
- `borrowApi.getAdminPage(...)`

这说明一个很典型的前端实践：

**一个页面经常不是只对应一个接口，而是要组合多个后端能力，拼出真正适合用户使用的界面。**

## 8. 视觉与交互风格设计

这次前端我没有走“默认白底灰表格”的安全路线，而是给它定了一个更明确的风格：

- 背景：暖米色
- 主色：深海军蓝
- 强调色：橙色
- 成功色：草地绿

这个方向对应的代码主要在：

- [app.css](/D:/newdir/sports-equipment-architecture-project/frontend/assets/css/app.css)

### 为什么这样做

因为这是一个“校园体育器材管理系统”，如果界面完全像财务系统，会很无聊，也不利于记住页面结构。

我想要的是：

- 有管理系统的秩序感
- 也有一点运动场景的活力

所以我用了：

- 暖底色
- 深色侧栏
- 橙色动作按钮
- 轻微网格背景

让页面一眼就能区分出层次，而不是全都糊在一起。

## 10. 这一轮开发我最希望你学到什么

如果你后面想借这份日志一步步理解整个前端开发，我最希望你抓住这 5 件事：

1. **先定页面边界，再写页面内容。**
   不是一上来就写按钮，而是先知道学生页、管理员页分别承担什么任务。

2. **先抽公共层，再铺业务页。**
   登录态、角色守卫、导航壳层、API 调用不该复制到每个页面里。

3. **不要把页面直接绑死在真实后端上。**
   mock 模式不是偷懒，而是为了把前端开发节奏从联调问题里解放出来。

4. **一个页面通常会组合多个接口。**
   比如管理员借还页同时要读预约和借用记录，这很正常。

5. **开发过程中可以升级方案，但要知道为什么升级。**
   比如这次我就做了几次关键调整：
   - 从“页面各写各的”改成“统一 shell”
   - 从“静态假数据”改成“可流转的 mock 业务层”
   - 从“借出页 / 归还页分开”改成“借还工作台合并”

这些变化不是反复横跳，而是随着结构越来越清晰，主动把方案往更稳、更易维护的方向推。

## 11. 下一步如果继续推进

这轮前端开发先不考虑联调，所以后面最自然的下一步是：

1. 给后端补最小 CORS 或静态资源同源托管方案
2. 把 `config.js` 里的 `USE_MOCK` 切到 `false`
3. 逐页联调：
   - 先 `login`
   - 再 `equipment`
   - 再 `reservation`
   - 最后 `borrow`

到那一步时，这套前端结构基本不用推翻，只需要把 mock 数据源换成真实 API 即可。
