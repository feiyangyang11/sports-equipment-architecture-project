# 开发细分计划

## 1. 文档目的

本文件不是课程报告，而是项目开发阶段的执行清单。它解决四个问题：

1. 现在先做什么。
2. 为什么先做这个。
3. 这一阶段要产出什么。
4. 做到什么程度算完成。

本项目采用“后端先行、前端跟进、一步一验收”的开发方式，目标是不抢跑、不堆代码、每一步都形成可验证成果。

## 3. 总体开发原则

### 原则一：先做最小可运行闭环

不要一开始就写完整业务，而是先做到：

- 工程能编译
- 服务能启动
- 接口能访问

### 原则二：先稳定后端契约，再做前端页面

前端依赖接口。如果先写前端，后端字段一改，前端就要反复返工。因此顺序应为：

`数据库 -> 后端接口 -> 前端页面`

### 原则三：每一步都要可验收

每一阶段都必须有明确结果，比如：

- 服务启动成功
- `/health` 返回成功
- `GET /api/equipment` 能查到 seed 数据

### 原则四：每次只增加一层复杂度

开发顺序应从简单到复杂：

1. 空服务
2. 基础接口
3. 数据库读取
4. 登录鉴权
5. 业务流程状态流转

## 5. 推荐开发顺序

## 阶段 A：后端最小骨架

### 目标

建立一个能编译、能启动、能访问基础接口的后端程序。

### 为什么先做

这是整个系统的基础。如果这一步没通，后续数据库、接口、前端都无从谈起。

### 预计产出

- `backend/CMakeLists.txt`
- `backend/src/app/` 下的应用入口代码
- 基础配置文件
- 一个最简单的健康检查接口，例如 `/health`

### 验收标准

- 项目能成功编译
- 启动后端服务不报错
- 浏览器或 `curl` 访问 `/health` 返回成功响应

### 学习重点

- 工程骨架如何搭建
- 程序入口和模块目录如何协作
- 什么叫“最小可运行系统”

## 阶段 B：数据库接入

### 目标

让后端程序连接 MySQL，并执行最简单的查询。

### 为什么第二步做

数据库脚本已经准备好，现在需要验证后端和数据库之间的通路，而不是立刻写复杂业务。

### 预计产出

- 数据库连接配置
- 连接初始化代码
- 一个简单查询示例

### 验收标准

- 后端启动时能成功连接数据库
- 能执行一条简单 SQL，如查询器材数量或分类数量
- 查询失败时有清晰错误日志

### 学习重点

- 配置与代码分离
- 数据访问和业务逻辑为什么不能混在一起
- 程序如何从“能跑”变成“能读数据”

## 阶段 C：第一个真实业务接口

### 目标

实现器材列表查询接口，建议优先做：

`GET /api/equipment`

### 为什么先做这个接口

器材查询是只读接口，风险小，最适合验证完整分层：

`Controller -> Service -> Repository -> Database`

### 预计产出

- `EquipmentController`
- `EquipmentService`
- `EquipmentRepository`
- `EquipmentDTO`

### 验收标准

- 能返回数据库中的器材列表
- 返回字段结构稳定
- 至少支持基础列表查询
- 最好支持按名称或状态筛选

### 学习重点

- 一次 HTTP 请求如何贯穿整个分层结构
- DTO、Model、Repository、Service 的职责边界
- 为什么查询接口适合作为第一条业务链路

## 阶段 D：登录接口

### 目标

实现用户登录接口，能区分学生和管理员。

### 为什么放在器材查询之后

登录涉及密码校验、角色判断、错误处理，复杂度比只读查询更高。先做查询接口能先把工程结构练熟。

### 预计产出

- `UserController`
- `UserService`
- `UserRepository`
- 登录请求/响应 DTO

### 验收标准

- 能按用户名查到用户
- 能校验密码
- 能返回用户角色、基础信息
- 错误用户名或错误密码能正确提示

### 学习重点

- 鉴权入口如何设计
- 为什么“登录成功返回什么”也是接口设计的一部分

## 阶段 E：预约主链路

### 目标

逐步打通以下链路：

1. 提交预约
2. 审核预约
3. 办理借出
4. 办理归还

### 为什么这是核心阶段

这部分最能体现软件开发中“业务规则 + 状态流转 + 数据一致性”的复杂性，也是本项目的核心价值所在。

### 预计产出

- `ReservationController / Service / Repository`
- `BorrowController / Service / Repository`
- 事务处理逻辑
- 状态校验逻辑

### 验收标准

- 学生可以提交预约
- 管理员可以审核预约
- 审核通过后可以办理借出
- 归还后库存会回收
- 状态迁移符合设计文档

### 学习重点

- 为什么复杂业务不能只靠前端校验
- 为什么事务边界很关键
- 为什么状态机思想能降低系统混乱

## 阶段 F：逾期处理与日志预留

### 目标

补齐逾期处理；操作日志能力暂不在本版本实现，保留后续添加空间。

### 预计产出

- 逾期处理相关逻辑
- 操作日志扩展预留说明

### 验收标准

- 能标记或跟踪逾期记录
- 文档中已明确操作日志为后续扩展项

### 学习重点

- 非主流程模块如何和主流程解耦
- 可审计性为什么是管理系统的重要质量属性

## 阶段 G：前端页面

### 目标

在后端基础接口稳定后，再开发前端页面。

### 页面优先级

1. 登录页
2. 学生器材查询与预约页
3. 管理员审核页
4. 借出归还处理页

### 验收标准

- 页面能调通已有后端接口
- 主要业务流程可演示

### 学习重点

- 前端页面为什么要依赖稳定接口契约
- 前后端联调如何发现问题

## 8. 当前风险与注意事项

### 风险一：前端抢跑

如果后端接口还没稳定就开始写页面，返工概率很高。

### 风险二：业务逻辑写进 Controller

一开始图快很容易把 SQL、业务判断、HTTP 返回揉在一起，后面会很难维护。

### 风险三：库存一致性只靠前端

预约、借出、归还必须由后端做最终校验，不能只靠页面按钮控制。

### 风险四：一步写太多

如果第一阶段就试图把登录、查询、预约一起做完，排错会非常困难。
